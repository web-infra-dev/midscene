# 编写和运行 YAML 测试用例

框架维护者完成 Test Project 配置后，人类或 Agent 可以使用项目注册的 Node 编写 YAML 测试用例。开始编写前，建议运行 `describe-nodes` 生成 Node Reference，确认项目提供的能力和输入参数。

如果你尚不了解整体设计，请先阅读 [YAML 测试运行器概览](./yaml-test-runner-overview)。如果项目尚未配置，请阅读[扩展和维护 YAML 测试运行器](./extend-yaml-test-runner)。

## 核心概念

`@midscene/test` 使用以下五个核心概念描述测试项目。

```text
Test Project
└── Workflow Document
    ├── Lifecycle Step
    └── Case
        └── Step
            └── Node
```

- **Node（节点）**：平台开发者注册的执行能力，例如 `aiAssert` 或 `order.create`。
- **Step（步骤）**：在 YAML 中对 Node 的一次调用。
- **Case（用例）**：由多个 Step 组成的具名测试用例。
- **Workflow Document（工作流文档）**：包含生命周期和多个 Case 的 YAML 文件。
- **Test Project（测试项目）**：项目配置，以及由该配置管理的一组 Workflow Document。

Node 定义团队可用的执行能力，YAML 定义每个测试用例如何组合这些能力。

## 编写 YAML 测试用例

每个 Workflow Document 必须包含非空的 `cases` 数组。每个 Case 必须包含 `name` 和非空的 `steps`。

```yaml
cases:
  - name: 创建订单
    steps:
      - order.create:
          sku: midscene-mug
          quantity: 1
      - aiAssert: 页面显示“下单成功”

  - name: 取消订单
    steps:
      - order.cancel:
          orderId: example-order-id
      - aiAssert: 页面显示“订单已取消”
```

运行器按照 YAML 中的声明顺序执行 Case。一个 Case 失败后，运行器会记录失败结果，并继续执行后续 Case。

### 编写 Step

每个 Step 只能调用一个 Node。如果调用时只需传入 `prompt` 参数，可以使用字符串简写。

```yaml
steps:
  - aiAct: 点击提交订单按钮
```

上面的写法等价于：

```yaml
steps:
  - aiAct:
      prompt: 点击提交订单按钮
```

业务 Node 可以接收自定义参数。

```yaml
steps:
  - order.create:
      sku: midscene-mug
      quantity: 2
```

### 设置超时和错误处理

`$` 用于设置由运行器控制的 Step 参数。运行器不会将这些参数传入 Node 的 `input`。

```yaml
steps:
  - order.create:
      sku: midscene-mug
      $:
        timeout: 30000
        continue-on-error: true
```

支持以下两个字段：

- `timeout`：Step 的超时时间，单位为毫秒。
- `continue-on-error`：设为 `true` 后，即使 Step 失败，运行器也会继续执行当前阶段的后续 Step。默认值为 `false`。

`continue-on-error` 只控制运行器是否继续执行。只要有 Step 失败，Case 的最终状态就是 `failed`。

## 定义执行生命周期

Workflow Document 可以在 `cases` 前后声明生命周期 Step。

```yaml
beforeAll:
  - data.prepare: 创建本文件需要的测试数据

beforeEach:
  - browser.reset: 将页面恢复到初始状态

cases:
  - name: 创建订单
    steps:
      - aiAct: 创建一个订单
      - aiAssert: 页面显示“下单成功”

  - name: 取消订单
    steps:
      - aiAct: 取消最新订单
      - aiAssert: 页面显示“订单已取消”

afterEach:
  - report.save: 保存当前用例的执行信息

afterAll:
  - data.cleanup: 删除本文件创建的测试数据
```

完整执行顺序如下。

```text
setupDocument
beforeAll
  Case 1：beforeEach → steps → afterEach
  Case 2：beforeEach → steps → afterEach
afterAll
文档清理阶段
```

各阶段的职责如下：

- `setupDocument()` 和文档清理阶段负责创建和释放运行资源。
- `beforeAll` 和 `afterAll` 是在每个 YAML 文件中执行一次的业务 Step。
- `beforeEach` 和 `afterEach` 是在每个 Case 中执行一次的业务 Step。

即使 Case 主体失败，`afterEach` 仍会执行。

如果 `beforeAll` 失败，运行器会将当前文件中的 Case 标记为 `not-run`，但仍会执行 `afterAll` 和清理函数。

如果 `setupDocument()` 失败，运行器会跳过 YAML 生命周期和 Case，但仍会运行已经注册的清理函数。

## 运行项目

运行当前目录中的 Test Project：

```bash
pnpm exec midscene-test
```

运行指定目录中的项目：

```bash
pnpm exec midscene-test ./e2e
```

指定配置文件或结果目录：

```bash
pnpm exec midscene-test \
  --config ./config/midscene.config.ts \
  --result-dir ./test-results
```

运行器会依次执行 Workflow Document 和 Case。当前版本仅支持串行执行，也不会自动重试失败的 Case。

### 查看运行结果

每次运行都会生成 `project.json`。详细结果按照类型保存在以下目录：

```text
test-results/
├── project.json
├── runs/
│   └── <case-id>/<run-id>.json
├── documents/
│   └── <document-id>/<document-run-id>.json
└── collection-errors/
    └── <error-id>.json
```

`project.json` 包含以下信息：

- 文件选择规则和本次找到的 YAML 文件。
- Case 的 `success`、`failed` 或 `not-run` 状态。
- Workflow Document 的生命周期状态。
- YAML 收集错误及对应的错误文件。
- 本次运行的汇总和 CLI 退出码。

Case 失败或未运行、Document 失败或发生收集错误时，CLI 会返回退出码 `1`。

## 当前限制

当前版本暂不支持以下能力：

- Case 或 Workflow Document 并发执行。
- 失败 Case 自动重试。
- DAG、分支、循环或 Step 并行。
- Workflow Document 之间的数据依赖。
- 从远程 URL、npm 包或 Git 仓库加载 YAML。

运行器会在执行前收集并校验所有 YAML 文件。未知的顶层字段、未知 Node、空 `cases` 或无效 Step 都会产生收集错误（collection error）。运行器会立即报告这些错误，不会等到执行相应 Case 时才报告。
