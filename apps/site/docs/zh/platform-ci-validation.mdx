# 在 CI 中验证 Midscene 的跨平台能力

本文说明 Midscene 如何通过持续集成（CI）验证 Web、Linux、Windows、macOS、Android 和 iOS。你可以通过本文找到平台 workflow、成功任务和对应的 Midscene HTML 报告。

## 对比各平台的 CI 覆盖范围

每条 workflow 都会通过 runner 桌面、模拟器或浏览器验证对应的平台适配器。workflow 会结合确定性检查与模型驱动的 TodoMVC 测试。

| 平台 | Workflow 与 runner | UI 环境 | 验证内容 |
| --- | --- | --- | --- |
| Web | [AI - Playwright e2e](https://github.com/web-infra-dev/midscene/blob/main/.github/workflows/ai.yml)，`ubuntu-22.04` | Playwright 驱动的桌面版 Chrome | TodoMVC 操作与查询、缓存回放、报告生成 |
| 无头 Linux | [Headless Linux](https://github.com/web-infra-dev/midscene/blob/main/.github/workflows/headless-linux.yml)，`ubuntu-22.04` | X 虚拟帧缓冲区（Xvfb）中的 Google Chrome | 在没有物理显示器的环境中执行 Computer TodoMVC 与 Chrome 扩展流程 |
| Windows | [Windows Desktop](https://github.com/web-infra-dev/midscene/blob/main/.github/workflows/windows-desktop.yml)，`windows-2022` | 可见的 WinForms fixture 与 Chrome TodoMVC | 包产物、单元测试、原生输入、截图和 TodoMVC |
| macOS | [macOS Desktop](https://github.com/web-infra-dev/midscene/blob/main/.github/workflows/macos-desktop.yml)，`macos-15-intel` | 可见的 AppKit fixture 与 Safari TodoMVC | 包产物、单元测试、原生输入、截图和 TodoMVC |
| Android | [Android Emulator](https://github.com/web-infra-dev/midscene/blob/main/.github/workflows/android-emulator.yml)，`ubuntu-22.04` | Android API 35 Pixel 6 模拟器 | 包产物、单元测试、Android 调试桥输入、截图和 TodoMVC |
| iOS | [iOS Simulator](https://github.com/web-infra-dev/midscene/blob/main/.github/workflows/ios-simulator.yml)，`macos-15` | 最新可用的 iPhone Simulator、Mobile Safari 与 WebDriverAgent | 包产物、单元测试、模拟器输入、截图和 TodoMVC |
| HarmonyOS | 仓库中暂无 workflow | 未覆盖 | 在 HarmonyOS 设备或设备农场中验证 |

桌面与移动端 workflow 会在任一必要阶段失败时返回失败。检查范围包括构建产物、包入口、单元测试、确定性冒烟，以及适用平台的 TodoMVC。

## 使用报告与诊断材料定位问题

即使测试失败，平台 workflow 也会上传证据。先使用 HTML 报告回放 Midscene 操作，再使用诊断材料检查平台状态。

| 平台 | 冒烟报告 | TodoMVC 报告 | 诊断材料 |
| --- | --- | --- | --- |
| Windows | `windows-desktop-smoke-report` | `windows-todo-mvc-report` | `windows-desktop-diagnostics` |
| macOS | `macos-desktop-smoke-report` | `macos-todo-mvc-report` | `macos-desktop-diagnostics` |
| Android | `android-emulator-smoke-report` | `android-todo-mvc-report` | `android-emulator-diagnostics` |
| iOS | `ios-simulator-smoke-report` | `ios-todo-mvc-report` | `ios-simulator-diagnostics` |
| 无头 Linux | `headless-linux-ai-report` | 包含在 AI 报告中 | Workflow 日志 |
| Web | `e2e-report` | 包含在端到端报告中 | `e2e-cache-report` 与 `e2e-report-output` |

诊断材料可以包含 runner 元数据、截图、平台日志、包清单和步骤结果。具体文件由平台 workflow 决定。

## 了解 PR 自动触发范围

Web workflow 会在每个 PR 和每次推送到 `main` 时运行。其他平台 workflow 使用路径过滤器控制 runner 用量。

- **Windows 与 macOS**：监听 `packages/computer/**`、`packages/core/**`、`packages/shared/**`、报告输入和 workspace 配置
- **Android**：监听 `packages/android/**`、`packages/core/**`、`packages/shared/**`、报告输入和 workspace 配置
- **iOS**：监听 `packages/ios/**`、`packages/core/**`、`packages/shared/**`、报告输入和 workspace 配置
- **无头 Linux**：监听 Computer、Core、Shared、Chrome 扩展、报告输入和 workspace 配置

每条 workflow 也会监听自身的 YAML 文件。你可以从覆盖范围表格打开 workflow 源码，查看当前的完整路径列表。

## 手动运行平台 workflow

所有平台 workflow 都支持 `workflow_dispatch`。触发任务前，请先将分支推送到 `web-infra-dev/midscene`。

```bash
branch_name=your_branch_name

gh workflow run windows-desktop.yml \
  --repo web-infra-dev/midscene \
  --ref "$branch_name" \
  -f branch="$branch_name"

gh run list \
  --repo web-infra-dev/midscene \
  --workflow windows-desktop.yml \
  --branch "$branch_name" \
  --limit 1
```

你可以将 `windows-desktop.yml` 替换为 `macos-desktop.yml`、`android-emulator.yml`、`ios-simulator.yml`、`headless-linux.yml` 或 `ai.yml`。

模型驱动测试从仓库 secrets 读取 `MIDSCENE_MODEL_API_KEY`，并从仓库 variables 读取模型配置。GitHub 不会向不受信任的 fork workflow 传递 Actions secrets。

## 理解 CI 验证边界

这些 workflow 可以证明 Midscene 能在对应 hosted runner 中执行被测链路，但不能认证所有设备、显示、权限或远程会话配置。

- **Windows**：使用 96 DPI，即 100% 缩放的交互会话。不覆盖提权窗口、混合 DPI 或远程桌面协议（RDP）会话
- **macOS**：使用已启用辅助功能权限的 hosted 交互桌面。不覆盖所有 macOS 版本或权限策略
- **Android**：使用 Android API 35 的 x86_64 Pixel 6 模拟器，不能替代真机验证
- **iOS**：通过 WebDriverAgent 控制 iPhone Simulator 与 Mobile Safari，不能替代真机验证
- **无头 Linux**：使用 Xvfb 代替物理显示器。窗口管理器和图形驱动行为可能与工作站不同
- **Web**：通过 Playwright 使用桌面版 Chrome。其他浏览器需要单独验证
- **HarmonyOS**：仓库中暂无 CI 冒烟，发布前需要验证目标设备
