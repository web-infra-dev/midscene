# 集成到 Rstest

[Rstest](https://github.com/web-infra-dev/rstest) 是基于 Rspack 的高性能测试框架。`@midscene/rstest` 把 Midscene 暴露成原生 Rstest fixtures，浏览器启动、Agent 创建、HTML 报告生成都由 Midscene 自动处理。

```ts
import { expect, test } from '@midscene/rstest/playwright';

test('adds a todo', async ({ agent, page }) => {
  await page.goto('http://localhost:5173/');
  await agent.aiAct("type 'Study AI' in the input and press Enter");
  await agent.aiAssert('the list contains exactly one item: "Study AI"');
});
```

`@midscene/rstest` 提供：

- 类型化 fixtures。`agent`、`page`、`browser`、`agentForPage` 直接从 test context 解构。
- Lazy 浏览器生命周期。一个文件里如果没有 test 依赖 `browser`，浏览器不会被启动。AI 测试和纯 unit test 混在同一个文件时，只有 AI 测试会产生 AI 开销。
- 可组合。用 `test.extend({ ... })` 覆盖 options 或任何单独的 fixture，比如预导航并注入 route 的 `page`，或者第二个 `adminAgent` persona。
- 每个文件一份合并后的 HTML 报告，路径在文件结束时打印到控制台。

这个包构建在 Rstest 官方 Playwright 集成 [`@rstest/playwright`](https://github.com/web-infra-dev/rstest/tree/main/packages/playwright) 之上。Midscene 在其上叠加 `agent` lifecycle 与报告能力，上游的全套工具箱直接继承：Playwright 风味的 `expect`（`toBeVisible`、`toHaveText` 等）、debug 模式（`PWDEBUG`、失败时暂停）、Playwright trace 采集，以及 `request` / `serve` fixtures。

:::info 目前仅支持 Playwright

目前只支持 Playwright 一种浏览器引擎，因此引擎显式写在 import 路径里（`@midscene/rstest/playwright`），与 `@midscene/web` 的 `/playwright`、`/puppeteer` 入口保持一致。暂不支持 Puppeteer。

:::

## 第一步：安装依赖

将 `@midscene/rstest` 与 `@midscene/web`、`@rstest/core`、`@rstest/playwright`、`playwright` 一起安装。要求 `@rstest/core >= 0.11.2`、`playwright >= 1.49.1`，以及 Node `^20.19.0 || >=22.12.0`。

<PackageManagerTabs command="install @midscene/web @midscene/rstest @rstest/core @rstest/playwright playwright --save-dev" />

## 第二步：配置 Rstest

在测试项目根目录新建 `rstest.config.ts`：

```ts title="rstest.config.ts"
import 'dotenv/config'; // 从 .env 加载模型相关环境变量
import MidsceneReporter from '@midscene/rstest/reporter';
import { defineConfig } from '@rstest/core';

export default defineConfig({
  include: ['e2e/**/*.test.ts'],
  // AI 调用较慢，testTimeout 与 hookTimeout 都留足。
  testTimeout: 1_800_000,
  hookTimeout: 60_000,
  reporters: ['default', new MidsceneReporter()],
});
```

`MidsceneReporter` 会把一个文件里所有测试的报告合并成一份 HTML，并打印形如 `Midscene report: midscene_run/report/<file>.html` 的一行。在浏览器中打开这个 HTML，即可逐步回放每个 AI 步骤与对应截图。合并报告依赖这个 reporter：不注册它，每个测试仍会各自生成报告，但不会有人把它们合并起来。

## 第三步：编写测试

```ts title="e2e/todo-list.test.ts"
import { expect, test } from '@midscene/rstest/playwright';

test('adds and completes a todo', async ({ agent, page }) => {
  await page.goto('http://localhost:5173/');
  await agent.aiAct("type 'Study AI' in the input and press Enter");
  await agent.aiAct('click the checkbox of the first item');

  const list = await agent.aiQuery<string[]>(
    'string[], the complete task list',
  );
  expect(list).toHaveLength(1);

  await agent.aiAssert('the only item is marked as done');
});
```

`@midscene/rstest/playwright` 导出的 `test` 是预先挂好了 Midscene fixtures 的 Rstest test。直接从 test context 解构需要的 fixture，fixture 引擎会按依赖关系拓扑展开 setup 与 teardown。

可用的 fixtures：

- **`agent`**：绑定到主 page 上的 Midscene `PlaywrightAgent`。AI action 和 assertion 的默认入口。导航是测试自己的事（`await page.goto(...)`），与 Midscene 的 Playwright 集成完全一致。
- **`page`**：Playwright `Page`，原样来自 `@rstest/playwright`——包括其失败时的行为（`PWDEBUG` 下失败保留现场并 pause）。用于 `agent` 做不到或不该做的事，比如 `page.route(...)` network mock、`page.evaluate(...)`、`page.context().cookies()`。
- **`browser`**：共享的 `Browser`，由 `@rstest/playwright` 管理，惰性启动、worker 内共享、空闲时自动关闭。适合在测试中开新 context 或新 page，用于多用户、跨会话场景。
- **`context`**：创建主 `page` 的 test-scoped `BrowserContext`。需要自定义 context 时通过 `test.extend` 覆盖。
- **`request`** / **`serve`**：继承自 `@rstest/playwright`。隔离的 `APIRequestContext` 用于 API 调用，以及一行代码起静态文件服务器，自动清理。
- **`agentForPage(page, opts?)`**：为另一个 page 构造 Midscene Agent，比如 popup 或手动开的 page。报告会和主 agent 合并，销毁自动完成。
- **`playwright`**：浏览器级配置（launch options、context options、debug、trace），原样来自 `@rstest/playwright`。详见 [浏览器配置](#浏览器配置playwright-fixture)。
- **`agentOptions`**：本测试构造的所有 agent 的默认选项。详见 [选项](#选项) 与 [项目级默认配置](#项目级默认配置可选)。

```ts
test('mocks the API and verifies the UI', async ({ agent, page }) => {
  await page.route('**/api/contacts', (route) =>
    route.fulfill({ json: [{ name: 'Fake Alice' }] }),
  );
  await page.goto('http://localhost:5173/');
  await agent.aiAssert('the contact list shows "Fake Alice"');
});

test('drives a popup with a second agent', async ({
  agent,
  page,
  agentForPage,
}) => {
  await page.goto('http://localhost:5173/');
  await agent.aiTap("the 'View details' button");
  const popup = await page.waitForEvent('popup');

  const popupAgent = await agentForPage(popup);
  await popupAgent.aiAssert('the popup shows order details');
});

test('runs two isolated user sessions in one test', async ({
  browser,
  agentForPage,
}) => {
  const sessionB = await browser.newContext({ storageState: 'user-b.json' });
  const pageB = await sessionB.newPage();
  await pageB.goto('http://localhost:5173/');
  const agentB = await agentForPage(pageB);

  await agentB.aiAct('send a message to user A');
});
```

## 第四步：运行测试

```bash
npx rstest run

# 调试时也可以用 watch 模式：
npx rstest watch
```

每个文件运行完后会输出：

```text
Midscene report: midscene_run/report/<file>.html
```

报告默认输出到 `midscene_run/report/`。详见[消费报告文件](./consume-report-file)。

## 项目级默认配置（可选）

如果希望所有文件共享同一份配置，在一个共享模块里扩展 base test 并导出，所有测试文件都从这里 import——与 Playwright Test 的模式完全一致：

```ts title="tests/midscene.ts"
import {
  test as base,
  defaultPlaywrightOptions,
} from '@midscene/rstest/playwright';

export { describe, expect } from '@midscene/rstest/playwright';

export const test = base.extend({
  playwright: {
    ...defaultPlaywrightOptions,
    contextOptions: { locale: 'zh-CN', viewport: { width: 1280, height: 720 } },
  },
  agentOptions: { cache: true },
});
```

```ts title="tests/todo.test.ts"
import { expect, test } from './midscene';
```

需要不同配置的文件再次扩展即可——后调用的 `extend` 覆盖先前的值（整体替换，标准 fixture 语义）。

## 配置 Midscene

Midscene 从四个渠道读取配置，按你想影响的作用域选一个。

1. **环境变量**（模型端点、API Key、vision-family 开关、cache 目录、report 目录、debug 开关）：用你平时设环境变量的方式即可（`.env`、CI secret、shell export）。Midscene runtime 会自动读取，不用额外接线。完整列表见 [模型配置](./model-config)。示例：

   ```bash
   MIDSCENE_MODEL_BASE_URL=https://api.openai.com/v1
   MIDSCENE_MODEL_API_KEY=sk-...
   MIDSCENE_MODEL_NAME=qwen3.7-plus
   MIDSCENE_DEBUG_MODEL_RESPONSE=1   # 把模型原始响应 dump 到磁盘
   MIDSCENE_RUN_DIR=./midscene_run   # 报告 + 缓存的根目录
   ```

2. **`agentOptions` fixture**：每个文件可独立配置，针对 per-agent 配置。接收完整 `AgentOpt`（`modelConfig`、`aiActionContext`、`cache`、`replanningCycleLimit`、`waitAfterAction`、`screenshotShrinkFactor`、`onTaskStartTip`、`onOpenAIClientCreated` 等）。`groupName` 和 `reportFileName` 由 lifecycle 保留。

   ```ts
   const test = base.extend({
     agentOptions: {
       aiActionContext: 'You are testing a multilingual checkout flow.',
       cache: { strategy: 'read-write' },
       waitAfterAction: 500,
     },
   });
   ```

3. **`overrideAIConfig(...)`**：在 setup file 里用，做环境变量表达不了的运行时全局覆盖（比如根据是否 CI 切模型）。已从 `@midscene/rstest/playwright` re-export。

   ```ts title="midscene.setup.ts"
   import { overrideAIConfig } from '@midscene/rstest/playwright';

   overrideAIConfig({
     MIDSCENE_MODEL_NAME: process.env.CI ? 'gpt-5.5' : 'gpt-5.4',
   });
   ```

4. **Debug 日志**：`@midscene/rstest` 用 `debug` 包，按需通过 `DEBUG` 环境变量打开 topic：

   ```bash
   DEBUG=midscene:*,rstest:* npx rstest run
   ```

## 选项

配置分为两个互不重叠的域，各自只有一个入口：

- **浏览器域** → `playwright` fixture，与 `@rstest/playwright` 完全一致。
- **Midscene 域** → `agentOptions` fixture。

### 浏览器配置（`playwright` fixture）

launch options、context options、浏览器选择、debug、trace 全部配置在 `@rstest/playwright` 自己的 `playwright` fixture 上——[上游文档](https://github.com/web-infra-dev/rstest/tree/main/packages/playwright)里的每一个选项都原样可用：

```ts
const test = base.extend({
  playwright: {
    launchOptions: { headless: false },
    contextOptions: { locale: 'zh-CN', storageState: 'auth.json' },
    debug: true, // headed + slowMo + DevTools，失败时暂停（也可用 PWDEBUG=1）
    trace: 'retain-on-failure', // 保存 trace.zip（也可用 RSTEST_PLAYWRIGHT_TRACE）
  },
});
```

`@midscene/rstest` 在这里做的唯一一件事是提供一份默认值，以 `defaultPlaywrightOptions` 导出：

```ts
{
  launchOptions: {
    headless: /* CI 中 true，本地 false */,
    args: ['--no-sandbox', '--ignore-certificate-errors'],
  },
  contextOptions: { viewport: { width: 1920, height: 1080 } },
}
```

覆盖 `playwright` fixture 时整个对象被替换——标准 fixture 语义，没有任何隐式合并。想保留默认值就显式 spread：

```ts
import {
  test as base,
  defaultPlaywrightOptions,
} from '@midscene/rstest/playwright';

const test = base.extend({
  playwright: {
    ...defaultPlaywrightOptions,
    contextOptions: { viewport: { width: 375, height: 667 } },
  },
});
```

trace 产物落在 `.rstest/playwright-traces/`，附带 AI 可读的 `trace-summary.json`，控制台会打印 `npx playwright show-trace ...` 命令。

### Midscene 配置（`agentOptions`）

`agentOptions` fixture 承载本测试构造的所有 agent 的默认选项，原样透传给 `PlaywrightAgent`（如 `aiActionContext`、`modelConfig`、`waitAfterAction`）——Midscene 上游新增什么选项就自动流过什么，本包无需改动。`groupName` 与 `reportFileName` 由 lifecycle 管理。`cache` 接受 `true | false | { strategy?, id? }`；当 `cache: true` 或 `id` 缺省时，会自动从测试文件名 + test 名派生一个稳定 id，让重复跑同一个测试复用同一个 cache namespace。`agentForPage(page, opts)` 的 per-call `opts` 按 top-level 键覆盖这里的值。

```ts
const test = base.extend({
  agentOptions: { cache: true, aiActionContext: '先关掉任何弹出的促销对话框。' },
});
```

## 自定义 page 设置

想让整个文件的测试都落在同一个 URL 上，或默认 lifecycle 无法满足的场景（per-test storage state、init script、CDP 接线），直接覆盖 `page` fixture——这就是普通的 `@rstest/playwright` fixture 组合，`@midscene/rstest` 没有在上游 `page` 之上加任何东西。`await use(page)` 之前是 setup，之后是 teardown。

```ts
import { test as base } from '@midscene/rstest/playwright';

const test = base.extend({
  page: async ({ context }, use) => {
    await context.addInitScript(() => {
      window.localStorage.setItem('feature-flag', 'on');
    });
    const page = await context.newPage();
    await page.goto('https://example.com');

    await use(page);

    await page.close();
  },
});
```

`page` fixture 的 teardown 在 `agent.destroy()` **之后**触发，此时 page 仍然存活，这正是保存录屏、dump cookies 等操作的窗口期。共享的 `browser` 和默认构造的 `context` 也都还活着；只有你自己开的资源由你自己的 teardown 决定何时关闭。

如果需要更深层定制（比如完全自建一个 persistent context），覆盖 `context` fixture，默认的 `page` 会自动复用你产出的 context。注意 Rstest 的 fixture 覆写不能引用被覆写的原值：自定义 `page` 从 `context` 构建，自定义 `context` 从 `browser` 构建。

## 多 Persona 与组合 fixtures

`test.extend` 是可组合的，像叠 options 一样叠 fixtures：

```ts
const test = base.extend<{
  adminAgent: PlaywrightAgent;
}>({
  adminAgent: async ({ browser }, use) => {
    const context = await browser.newContext({ storageState: 'admin.json' });
    const page = await context.newPage();
    await page.goto('http://localhost:5173/');
    const agent = new PlaywrightAgent(page);
    await use(agent);
    await agent.destroy();
    await context.close();
  },
});

test('only admin can delete', async ({ agent, page, adminAgent }) => {
  await page.goto('http://localhost:5173/');
  await agent.aiAssert('no Delete button is visible');
  await adminAgent.aiAssert('a Delete button is visible');
});
```

这是多 persona 测试、mock server fixture、数据库 seed 以及其它你过去写在 `beforeEach` 里的东西的推荐写法。fixtures 按依赖图组合，按反向顺序 teardown，顺序显式且确定。
