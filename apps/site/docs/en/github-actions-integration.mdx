# Integrate Midscene with GitHub Actions

This guide shows you how to add Midscene checks for Web, headless Linux, Windows, macOS, Android, and iOS to GitHub Actions. Use the Midscene repository workflows as templates, configure GitHub repository secrets and variables, then collect HTML reports and diagnostics as GitHub artifacts.

In this guide, GitHub CI means GitHub Actions. Its workflow files, GitHub-hosted runners, secrets, variables, artifacts, and `gh` commands do not apply directly to another CI provider.

## Add Midscene to a GitHub repository

Start with one platform workflow, then adapt its install, build, and test commands to your repository. The linked workflows test the Midscene monorepo, so they are integration examples rather than files you can copy without changes.

1. Copy the platform workflow into `.github/workflows/` and keep its `pull_request` and `workflow_dispatch` triggers.
2. Add `MIDSCENE_MODEL_API_KEY` as a GitHub Actions repository secret.
3. Add `MIDSCENE_MODEL_BASE_URL`, `MIDSCENE_MODEL_NAME`, and `MIDSCENE_MODEL_FAMILY` as GitHub Actions repository variables.
4. Update `pull_request.paths` to include the platform package and every shared package that can affect it.
5. Keep the `actions/upload-artifact` steps so successful and failed jobs both retain reports and diagnostics.
6. Push the branch, open the **Actions** tab, and inspect the platform job and its artifacts.

GitHub does not pass repository Actions secrets to workflows triggered from untrusted forks. Run model-backed checks from a trusted branch or use a reviewed relay branch in the main repository.

GitHub may require you to sign in before downloading an artifact. The desktop and mobile workflows retain artifacts for seven days, so an artifact can expire while its run remains visible.

Download an artifact with the GitHub CLI while it remains available:

```bash
run_id=29252027901
artifact_name=windows-todo-mvc-report

gh run download "$run_id" \
  --repo web-infra-dev/midscene \
  --name "$artifact_name"
```

If an artifact expires, open the workflow from the platform table and select its latest successful run. The **Artifacts** section contains current reports and diagnostics.

## Upload reports and diagnostics as GitHub artifacts

The platform workflows upload evidence even when a job fails. Use the HTML report to replay Midscene actions, then use the diagnostics artifact to inspect the runner or device state.

| Platform | Smoke report | TodoMVC report | Diagnostics |
| --- | --- | --- | --- |
| Windows | `windows-desktop-smoke-report` | `windows-todo-mvc-report` | `windows-desktop-diagnostics` |
| macOS | `macos-desktop-smoke-report` | `macos-todo-mvc-report` | `macos-desktop-diagnostics` |
| Android | `android-emulator-smoke-report` | `android-todo-mvc-report` | `android-emulator-diagnostics` |
| iOS | `ios-simulator-smoke-report` | `ios-todo-mvc-report` | `ios-simulator-diagnostics` |
| Headless Linux | `headless-linux-ai-report` | Included in the AI report | Workflow logs |
| Web | `e2e-report` | Included in the end-to-end report | `e2e-cache-report` and `e2e-report-output` |

Diagnostics can contain runner metadata, screenshots, platform logs, package manifests, and step outcomes. The exact files depend on the platform workflow.

## Configure pull request path filters

GitHub Actions evaluates `pull_request.paths` before it creates a platform job. Include the platform package and all shared build or runtime dependencies to avoid a coverage gap.

- **Windows and macOS**: Watch `packages/computer/**`, `packages/core/**`, `packages/shared/**`, report inputs, and workspace configuration
- **Android**: Watch `packages/android/**`, `packages/core/**`, `packages/shared/**`, report inputs, and workspace configuration
- **iOS**: Watch `packages/ios/**`, `packages/core/**`, `packages/shared/**`, report inputs, and workspace configuration
- **Headless Linux**: Watch Computer, Core, Shared, Chrome extension, report inputs, and workspace configuration

Each workflow also watches its own YAML file. Open the workflow source from the platform table for its current path list.

## Run a workflow with the GitHub CLI

Every platform workflow supports `workflow_dispatch`. Push the branch to GitHub before dispatching it.

```bash
branch_name=your_branch_name

gh workflow run windows-desktop.yml \
  --repo web-infra-dev/midscene \
  --ref "$branch_name" \
  -f branch="$branch_name"

gh run list \
  --repo web-infra-dev/midscene \
  --workflow windows-desktop.yml \
  --branch "$branch_name" \
  --limit 1
```

Replace `windows-desktop.yml` with `macos-desktop.yml`, `android-emulator.yml`, `ios-simulator.yml`, `headless-linux.yml`, or `ai.yml`.

## Understand the GitHub-hosted runner boundary

These workflows prove the tested paths on specific GitHub-hosted runners. They do not certify every device, display, permission, or remote-session configuration.

- **Windows**: Uses an interactive session at 96 dots per inch (DPI), which is 100% scaling. It does not cover elevated windows, mixed DPI, or Remote Desktop Protocol (RDP) sessions
- **macOS**: Uses the hosted interactive desktop with Accessibility enabled. It does not cover every macOS version or permission policy
- **Android**: Uses an x86_64 Pixel 6 emulator with Android API 35. It does not replace physical-device testing
- **iOS**: Uses an iPhone Simulator and Mobile Safari through WebDriverAgent. It does not replace physical-device testing
- **Headless Linux**: Uses Xvfb instead of a physical display. Window manager and graphics-driver behavior can differ from a workstation
- **Web**: Uses Desktop Chrome through Playwright. Other browsers require separate validation
- **HarmonyOS**: Has no GitHub Actions smoke. Validate the target device before shipping
