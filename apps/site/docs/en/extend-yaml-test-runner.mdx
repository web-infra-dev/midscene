# Extend and maintain a YAML test runner

`@midscene/test` lets framework maintainers register domain-specific Nodes, manage runtime resources, and define document lifecycles. Use these capabilities to connect browsers, Agents, external tools, and business APIs.

This guide covers building and maintaining the runner. For the overall design, start with the [YAML test runner overview](./yaml-test-runner-overview).

## Quick start

Install the runner and the integrations used by your project.

```bash
pnpm add -D @midscene/test @midscene/web playwright
```

Create this structure:

```text
team-test-runner/
├── cases/
│   └── midscene.yaml
└── midscene.config.ts
```

The project configuration must be named `midscene.config.ts`. It directly default-exports a Test Project definition.

```ts filename=midscene.config.ts
import { defineNode, z } from '@midscene/test';
import {
  defineProjectSetup,
  defineTestProject,
} from '@midscene/test/config';
import { createMidsceneNodes } from '@midscene/test/midscene';
import { PlaywrightAgent } from '@midscene/web/playwright';
import { chromium, type Browser, type Page } from 'playwright';

interface ProjectContext {
  browser: Browser;
}

interface DocumentContext extends ProjectContext {
  page: Page;
  agent?: PlaywrightAgent;
}

const setLanguageInputSchema = z.strictObject({
  language: z
    .enum(['en', 'zh'])
    .describe('The browser language used by the new Playwright Context.'),
});

const setLanguage = defineNode<
  typeof setLanguageInputSchema,
  unknown,
  DocumentContext
>({
  name: 'browser.setLanguage',
  description: 'Switch the browser language and reopen the current page.',
  inputSchema: setLanguageInputSchema,
  async execute({ input, context }) {
    const previousContext = context.page.context();
    const browserContext = await context.browser.newContext({
      locale: input.language === 'zh' ? 'zh-CN' : 'en-US',
    });
    context.page = await browserContext.newPage();
    await previousContext.close();
  },
});

const midsceneNodes = createMidsceneNodes<DocumentContext>({
  getAgent: ({ context }) => {
    context.agent ??= new PlaywrightAgent(context.page);
    return context.agent;
  },
});

const playwrightSetup = defineProjectSetup<ProjectContext>({
  name: 'playwright',
  platform: 'web',
  async setup({ onTeardown }) {
    const browser = await chromium.launch({ headless: true });
    onTeardown(() => browser.close());
    return { browser };
  },
});

export default defineTestProject<ProjectContext, DocumentContext>({
  projects: [
    {
      name: 'chromium',
      platform: 'web',
      setup: playwrightSetup,
      files: { include: ['cases/**/*.{yaml,yml}'] },
      tags: { include: ['smoke'], exclude: ['manual'] },
    },
  ],
  nodes: [setLanguage, ...midsceneNodes],

  async setupDocument({ projectContext, onTeardown }) {
    const browserContext = await projectContext.browser.newContext();
    onTeardown(() => browserContext.close());
    const page = await browserContext.newPage();
    await page.goto('https://midscenejs.com');
    return { browser: projectContext.browser, page };
  },
});
```

Add a YAML test document:

```yaml filename=cases/midscene.yaml
cases:
  - name: Midscene docs follow the browser language
    tags: [smoke]
    steps:
      - browser.setLanguage:
          language: en
      - aiAct: Click Documentation to open the Midscene docs
      - aiAssert:
          prompt: The page is the English Midscene documentation
          message: The documentation is not displayed in English
      - browser.setLanguage:
          language: zh
      - aiAssert:
          prompt: The page is the Chinese Midscene documentation
          message: The documentation is not displayed in Chinese
```

Run the project:

```bash
pnpm exec midscene-test
```

The CLI automatically loads `midscene.config.ts`, discovers matching YAML files, and runs their Cases serially. The default result directory is:

```text
.midscene/test-results/<timestamp>-<process-id>/
```

The stable public summary defaults to `midscene_run/output/summary.json`, while finalized Midscene reports default to `midscene_run/report/`.

## Define Node inputs with Zod

`inputSchema` is optional. When a Node declares it, the runner validates the YAML input with Zod before calling `execute()`. TypeScript infers `ctx.input` from the Schema, so you do not need to maintain a separate Input interface.

Import the compatible Zod instance from `@midscene/test` and prefer `z.strictObject()`. Strict objects reject unknown YAML fields. Field descriptions are also included in the generated Node Reference.

```ts
import { defineNode, z } from '@midscene/test';

const createOrderInputSchema = z.strictObject({
  sku: z.string().min(1).describe('The product SKU.'),
  quantity: z.number().int().positive().describe('The quantity to order.'),
});

const createOrder = defineNode({
  name: 'order.create',
  description: 'Create an order in the test environment.',
  inputSchema: createOrderInputSchema,
  async execute({ input }) {
    // input is inferred as { sku: string; quantity: number }.
    await createTestOrder(input);
  },
});
```

When a Node needs a typed Document Context, provide the Schema type, output type, and Context type:

```ts
defineNode<typeof createOrderInputSchema, { id: string }, ProjectContext>({
  name: 'order.create',
  description: 'Create an order in the test environment.',
  inputSchema: createOrderInputSchema,
  async execute({ input, context }) {
    return { data: await context.orders.create(input) };
  },
});
```

Without `inputSchema`, the runner preserves the existing behavior and passes the YAML input directly to the Handler. The Handler must validate it, and the generated reference cannot describe its fields.

## Use the Node execution context

Every Node receives the same execution contract:

- `input`: Node input after string shorthand normalization and variable resolution; `$` is removed.
- `$`: normalized runner metadata such as `timeoutMs` and `continueOnError`.
- `scope` plus `case` or `document`: the current identity, lifecycle phase, and Step index.
- `signal`: the Step timeout and cancellation signal.
- `context`: the value returned by `setupDocument()`.
- `history`: a deeply read-only, JSON-compatible view of completed Nodes.
- `onTeardown()`: register attempt- or document-scoped cleanup in LIFO order.

`history` contains completed `beforeAll` Nodes and the current Case attempt's completed Nodes. Each retry gets a fresh Case history. All Nodes receive history by default; deterministic Nodes may ignore it. The built-in `aiAct` and `aiAssert` Nodes merge it into Agent context, and the built-in `agent` Node passes it to the configured Agent executor.

A Node that owns an attempt-scoped resource should release it through `onTeardown()`, even when a later Step fails:

```ts
const sessionNode = defineNode({
  name: 'session.open',
  async execute({ onTeardown }) {
    const session = await openSession();
    onTeardown(async () => {
      await session.close();
      return { reportPaths: [session.reportPath] };
    });
  },
});
```

Returned report paths must be absolute. The runner stores them as paths relative to `summary.json`.

For retry isolation and finalized Midscene report indexing, prefer `agentProvider` over the compatibility `getAgent` callback:

```ts
const agents = new Map<string, PlaywrightAgent>();

const midsceneNodes = createMidsceneNodes<DocumentContext>({
  agentProvider: {
    getAgent(runId, { context }) {
      let agent = agents.get(runId);
      if (!agent) {
        agent = new PlaywrightAgent(context.page, {
          reportFileName: runId,
          cache: { id: runId },
        });
        agents.set(runId, agent);
      }
      return agent;
    },
    async releaseAgent(runId) {
      const agent = agents.get(runId);
      if (!agent) return;
      agents.delete(runId);
      await agent.destroy();
      return { reportPath: agent.reportFile ?? undefined };
    },
  },
});
```

The runner calls `releaseAgent()` exactly once after the attempt or Document scope that first requested the Agent. A retry has a new run ID, so it cannot reuse the previous Agent or cache scope.

## Configure a Test Project

`defineTestProject()` provides contextual types for the project fields and Document Context.

```ts filename=midscene.config.ts
export default defineTestProject<ProjectContext>({
  root: './e2e',
  files: {
    include: ['cases/**/*.{yaml,yml}'],
    exclude: ['cases/**/*.draft.yaml'],
  },
  projects: [
    {
      name: 'android-smoke',
      platform: 'android',
      setup: doraAndroidSetup,
      tags: { include: ['smoke'], exclude: ['manual'] },
      repeat: 2,
      retry: 1,
      variables: { appName: 'Example', reinstall: false },
    },
  ],
  testRunner: {
    maxConcurrency: 1,
    bail: 0,
    testTimeout: 120_000,
  },
  output: {
    summary: './midscene_run/output/summary.json',
    reportDir: './midscene_run/report',
  },
  nodes: [createOrder, ...midsceneNodes],
  async setupDocument({ projectContext, env, onTeardown }) {
    const database = await connectTestDatabase(env.TEST_DATABASE_URL);
    onTeardown(() => database.close());
    return { ...projectContext, database };
  },
});
```

Declare reusable Project setup with `defineProjectSetup()`. Setup runs once for the selected Execution Project and can acquire a browser, device lease, connection-renewal task, or other shared resource. Registered Project teardown functions run in LIFO order after every repeat and document has finished, including setup failure and interruption.

Each `projects[]` entry requires a unique `name` and a `platform` of `web`, `android`, `ios`, or `computer`. It may override `files`, configure `tags`, bind one setup definition, provide JSON-compatible `variables`, and set `repeat` and whole-Case `retry`. Root-level `files` remains as a compatibility default inherited by Projects that do not override it.

`testRunner.maxConcurrency` currently supports only `1`. `testRunner.bail` stops scheduling after the configured number of final Case failures, and `testRunner.testTimeout` is the default Step timeout.

The project root is selected in this order:

1. The `project-directory` CLI argument.
2. The `root` field in `midscene.config.ts`.
3. The current working directory.

`files.include` and `files.exclude` are glob patterns relative to the project root. The runner always ignores `.git`, `.midscene`, and `node_modules`.

Configuration files support only the `.ts` extension. `.js`, `.cjs`, `.mts`, `.cts`, and `.tsx` are rejected. Runtime loading does not resolve `tsconfig.json` `paths`; use relative imports with an explicit `.ts` extension or package specifiers.

## Generate a Node Reference

Generate a Markdown reference for every Node registered by the current Test Project:

```bash
pnpm exec midscene-test describe-nodes > midscene-nodes.md
```

You can also select a project directory or configuration file:

```bash
pnpm exec midscene-test describe-nodes ./e2e
pnpm exec midscene-test describe-nodes --config ./config/midscene.config.ts
```

The reference contains the common Step contract plus each Node name, description, and JSON Schema. The JSON Schema is generated from the Zod `inputSchema`. The built-in `aiAct`, `aiAssert`, `recordToReport`, `launch`, `wait`, and `agent` Nodes include their own schemas.

Nodes without `description` or `inputSchema` are still listed. The document marks the missing fields, and the CLI writes warnings to stderr instead of guessing their behavior.

`describe-nodes` only loads `midscene.config.ts`. It does not read Workflow YAML, run Project setup or `setupDocument()`, execute Node Handlers, or create a result directory. Add the generated `midscene-nodes.md` to an AI Agent's context so it can write YAML Cases using the capabilities that the project actually exposes.

## Programmatic API

Use these APIs when embedding the runner in another tool:

- `CaseRunner` or `createCaseRunner()` runs one object-form Case.
- `collectWorkflowDocument()` parses and statically validates one YAML file.
- `runWorkflowDocument()` runs one document lifecycle and its Cases.
- `createProjectRuntime()` manages one Execution Project's setup and teardown.
- `loadTestProject()` asynchronously loads a TypeScript project configuration.
- `runTestProject()` discovers, runs, and summarizes a Test Project.

The project-level APIs are exported from `@midscene/test/config`. Configuration loading is asynchronous; there is no synchronous loader.

After configuring the runner, continue with [Write and run YAML test cases](./yaml-test-cases) for YAML syntax, lifecycle behavior, and result handling.
