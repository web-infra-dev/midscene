# Integrate with Rstest

[Rstest](https://github.com/web-infra-dev/rstest) is a fast Rspack-based testing framework. With `@midscene/rstest` you can write AI-driven end-to-end tests as plain Rstest fixtures. Midscene launches the browser, creates an Agent per test, and produces an HTML report at the end of every test file.

```ts
import { expect, test as base } from '@midscene/rstest/playwright';

const test = base.extend({ url: 'http://localhost:5173/' });

test('adds a todo', async ({ agent }) => {
  await agent.aiAct("type 'Study AI' in the input and press Enter");
  await agent.aiAssert('the list contains exactly one item: "Study AI"');
});
```

`@midscene/rstest` provides:

- Typed fixtures. Destructure `agent`, `page`, `browser`, or `agentForPage` directly from the test context.
- A lazy browser lifecycle. If no test in a file depends on `browser`, no browser is launched, so a file that mixes AI tests with plain unit tests only pays the AI cost for the AI tests.
- Composable setup. Use `test.extend({ ... })` to override the URL, the options, or any individual fixture, such as a `page` that injects routes or a second `adminAgent` persona.
- One merged Midscene HTML report per test file, with its path printed to the console after the file finishes.

The package is built on [`@rstest/playwright`](https://github.com/web-infra-dev/rstest/tree/main/packages/playwright), Rstest's official Playwright integration. Midscene layers the `agent` lifecycle and reporting on top, and you inherit the upstream toolbox as-is: Playwright-flavored `expect` (`toBeVisible`, `toHaveText`, etc.), debug mode (`PWDEBUG`, pause on failure), Playwright trace capture, and the `request` / `serve` fixtures.

:::info Playwright only

Playwright is the only browser engine supported today, which is why the engine is explicit in the import path (`@midscene/rstest/playwright`), mirroring `@midscene/web`'s `/playwright` and `/puppeteer` entries. Puppeteer is not supported.

:::

## Step 1: Install dependencies

Install `@midscene/rstest` together with `@midscene/web`, `@rstest/core`, `@rstest/playwright`, and `playwright`. Requires `@rstest/core >= 0.11.2`, `playwright >= 1.49.1`, and Node `^20.19.0 || >=22.12.0`.

<PackageManagerTabs command="install @midscene/web @midscene/rstest @rstest/core @rstest/playwright playwright --save-dev" />

## Step 2: Configure Rstest

Create `rstest.config.ts` in the root of your test project:

```ts title="rstest.config.ts"
import 'dotenv/config'; // load model env vars from .env
import MidsceneReporter from '@midscene/rstest/reporter';
import { defineConfig } from '@rstest/core';

export default defineConfig({
  include: ['e2e/**/*.test.ts'],
  // AI calls are slow, so leave plenty of headroom for both tests and hooks.
  testTimeout: 1_800_000,
  hookTimeout: 60_000,
  reporters: ['default', new MidsceneReporter()],
});
```

`MidsceneReporter` merges the reports of every test in a file into one HTML report and prints a line like `Midscene report: midscene_run/report/<file>.html`. Open it in your browser to replay every AI step with screenshots. Registering the reporter is required for the merged report: without it each test still writes its own report, but nothing combines them.

## Step 3: Write a test

```ts title="e2e/todo-list.test.ts"
import { expect, test as base } from '@midscene/rstest/playwright';

const test = base.extend({ url: 'http://localhost:5173/' });

test('adds and completes a todo', async ({ agent }) => {
  await agent.aiAct("type 'Study AI' in the input and press Enter");
  await agent.aiAct('click the checkbox of the first item');

  const list = await agent.aiQuery<string[]>(
    'string[], the complete task list',
  );
  expect(list).toHaveLength(1);

  await agent.aiAssert('the only item is marked as done');
});
```

The `test` exported from `@midscene/rstest/playwright` is an Rstest test pre-wired with Midscene fixtures. Destructure what you need from the test context, and the fixture engine resolves dependencies and runs setup/teardown around each test.

The available fixtures are:

- **`agent`**: the Midscene `PlaywrightAgent` bound to the primary page. Default tool for AI actions and assertions. Its setup navigates `page` to `url` before the test body runs.
- **`page`**: the Playwright `Page`, straight from `@rstest/playwright` — including its failure behavior (kept open and paused under `PWDEBUG`). Escape hatch for things `agent` can't or shouldn't do, such as `page.route(...)` network mocking, `page.evaluate(...)`, or `page.context().cookies()`. In a test that doesn't use `agent`, navigate it yourself.
- **`browser`**: the shared `Browser`, managed by `@rstest/playwright`. It is launched lazily, shared per worker, and closed automatically when idle. Use it to spin up extra contexts or pages mid-test, for multi-user or cross-session scenarios.
- **`context`**: the test-scoped `BrowserContext` the primary `page` is created from. Override via `test.extend` if you need a custom context per test.
- **`request`** / **`serve`**: inherited from `@rstest/playwright`. An isolated `APIRequestContext` for API calls, and a one-liner static file server with automatic teardown.
- **`agentForPage(page, opts?)`**: build a Midscene agent for another page, such as a popup or a manually-created page. Its report is merged with the primary's, and destroy is automatic.
- **`playwright`**: browser-level configuration (launch options, context options, debug, trace), straight from `@rstest/playwright`. See [Browser configuration](#browser-configuration-the-playwright-fixture).
- **`url`** / **`midsceneOptions`**: Midscene-side configuration fixtures. See [Options](#options) and [Project-wide defaults](#project-wide-defaults-optional).

```ts
test('mocks the API and verifies the UI', async ({ agent, page }) => {
  await page.route('**/api/contacts', (route) =>
    route.fulfill({ json: [{ name: 'Fake Alice' }] }),
  );
  await agent.aiAssert('the contact list shows "Fake Alice"');
});

test('drives a popup with a second agent', async ({
  agent,
  page,
  agentForPage,
}) => {
  await agent.aiTap("the 'View details' button");
  const popup = await page.waitForEvent('popup');

  const popupAgent = await agentForPage(popup);
  await popupAgent.aiAssert('the popup shows order details');
});

test('runs two isolated user sessions in one test', async ({
  browser,
  agentForPage,
}) => {
  const sessionB = await browser.newContext({ storageState: 'user-b.json' });
  const pageB = await sessionB.newPage();
  await pageB.goto('http://localhost:5173/');
  const agentB = await agentForPage(pageB);

  await agentB.aiAct('send a message to user A');
});
```

## Step 4: Run the tests

```bash
npx rstest run

# or, while iterating:
npx rstest watch
```

After each file finishes you'll see something like:

```text
Midscene report: midscene_run/report/<file>.html
```

Reports are written to `midscene_run/report/`. See [Consume report files](./consume-report-file) for programmatic access.

## Project-wide defaults (optional)

For options that should apply to every file, extend the base test in a shared module and import your extended `test` everywhere — the same pattern as Playwright Test:

```ts title="tests/midscene.ts"
import {
  test as base,
  defaultPlaywrightOptions,
} from '@midscene/rstest/playwright';

export { describe, expect } from '@midscene/rstest/playwright';

export const test = base.extend({
  playwright: {
    ...defaultPlaywrightOptions,
    contextOptions: { locale: 'en-US', viewport: { width: 1280, height: 720 } },
  },
  midsceneOptions: {
    agentOptions: { cache: true },
  },
});
```

```ts title="tests/todo.test.ts"
import { expect, test as shared } from './midscene';

const test = shared.extend({ url: 'http://localhost:5173/' });
```

A file that needs different values extends again — later `extend` calls override earlier ones (whole-value replacement, standard fixture semantics).

## Configure Midscene

Midscene reads configuration from four channels. Pick the one that matches the scope you want.

1. **Environment variables** (model endpoint, API key, vision-family flags, cache dir, report dir, debug toggles): set them however you usually set env vars (`.env`, CI secrets, shell exports). They flow into the Midscene runtime automatically, with nothing extra to wire up. The full list lives in the [model configuration reference](./model-config). Quick sample:

   ```bash
   MIDSCENE_MODEL_BASE_URL=https://api.openai.com/v1
   MIDSCENE_MODEL_API_KEY=sk-...
   MIDSCENE_MODEL_NAME=qwen3.7-plus
   MIDSCENE_DEBUG_MODEL_RESPONSE=1   # dump raw model responses to disk
   MIDSCENE_RUN_DIR=./midscene_run   # where reports + cache live
   ```

2. **`midsceneOptions.agentOptions`**: for anything you set per agent. Accepts the full `AgentOpt` surface (`modelConfig`, `aiActionContext`, `cache`, `replanningCycleLimit`, `waitAfterAction`, `screenshotShrinkFactor`, `onTaskStartTip`, `onOpenAIClientCreated`, etc.). `groupName` and `reportFileName` are reserved by the lifecycle.

   ```ts
   const test = base.extend({
     url: 'http://localhost:5173/',
     midsceneOptions: {
       agentOptions: {
         aiActionContext: 'You are testing a multilingual checkout flow.',
         cache: { strategy: 'read-write' },
         waitAfterAction: 500,
       },
     },
   });
   ```

3. **`overrideAIConfig(...)`** in a setup file: for programmatic global overrides that env vars can't express, or for swapping config between test runs. Re-exported from `@midscene/rstest/playwright`.

   ```ts title="midscene.setup.ts"
   import { overrideAIConfig } from '@midscene/rstest/playwright';

   overrideAIConfig({
     MIDSCENE_MODEL_NAME: process.env.CI ? 'gpt-5.5' : 'gpt-5.4',
   });
   ```

4. **Debug logs**: `@midscene/rstest` uses the `debug` package; turn on what you want via the `DEBUG` env var.

   ```bash
   DEBUG=midscene:*,rstest:* npx rstest run
   ```

## Options

Configuration is split into two non-overlapping domains, each with exactly one home:

- **Browser domain** → the `playwright` fixture, exactly as in `@rstest/playwright`.
- **Midscene domain** → the `midsceneOptions` fixture.

### Browser configuration (the `playwright` fixture)

Launch options, context options, browser choice, debug, and trace are all configured on `@rstest/playwright`'s own `playwright` fixture — every option in the [upstream docs](https://github.com/web-infra-dev/rstest/tree/main/packages/playwright) works verbatim:

```ts
const test = base.extend({
  url: 'http://localhost:5173/',
  playwright: {
    launchOptions: { headless: false },
    contextOptions: { locale: 'zh-CN', storageState: 'auth.json' },
    debug: true, // headed + slowMo + DevTools, pause on failure (also: PWDEBUG=1)
    trace: 'retain-on-failure', // save trace.zip (also: RSTEST_PLAYWRIGHT_TRACE)
  },
});
```

The only thing `@midscene/rstest` adds here is a default value, exported as `defaultPlaywrightOptions`:

```ts
{
  launchOptions: {
    headless: /* true in CI, false locally */,
    args: ['--no-sandbox', '--ignore-certificate-errors'],
  },
  contextOptions: { viewport: { width: 1920, height: 1080 } },
}
```

Overriding the `playwright` fixture replaces the whole object — standard fixture semantics, no implicit merging. Spread the default when you want to keep it:

```ts
import {
  test as base,
  defaultPlaywrightOptions,
} from '@midscene/rstest/playwright';

const test = base.extend({
  url: 'http://localhost:5173/',
  playwright: {
    ...defaultPlaywrightOptions,
    contextOptions: { viewport: { width: 375, height: 667 } },
  },
});
```

Saved traces land in `.rstest/playwright-traces/` with an AI-readable `trace-summary.json` and a `npx playwright show-trace ...` command printed to the console.

### Midscene configuration (`midsceneOptions`)

| Field | Type | Default | Description |
| --- | --- | --- | --- |
| `gotoOptions` | `Parameters<Page['goto']>[1]` | — | Forwarded to `page.goto(url, ...)` when the `agent` fixture auto-navigates. |
| `agentOptions` | `WebPageAgentOpt` | — | Forwarded to `PlaywrightAgent` (e.g. `aiActionContext`, `modelConfig`). `groupName` and `reportFileName` are managed by the lifecycle. `cache` accepts `true \| false \| { strategy?, id? }`. When `cache: true` or `id` is omitted, a stable id is auto-derived from the test file name and test name so re-runs hit the same cache namespace. |

```ts
const test = base.extend({
  url: 'http://localhost:5173/',
  midsceneOptions: {
    agentOptions: { cache: true, aiActionContext: 'Close any promo dialog first.' },
  },
});
```

## Custom page setup

For lifecycles the defaults don't cover, such as per-test storage state, init scripts, or CDP wiring, override the `page` fixture — this is plain `@rstest/playwright` fixture composition; `@midscene/rstest` adds nothing on top of the upstream `page`. Code before `await use(page)` is setup; code after is teardown. The `agent` fixture still performs the `url` navigation on whatever page your override produces.

```ts
import { test as base } from '@midscene/rstest/playwright';

const test = base.extend({
  url: 'https://example.com',
  page: async ({ context }, use) => {
    await context.addInitScript(() => {
      window.localStorage.setItem('feature-flag', 'on');
    });
    const page = await context.newPage();

    await use(page);

    await page.close();
  },
});
```

Teardown of the `page` fixture runs **after** `agent.destroy()`, so the page is still alive at that point. That is exactly the window you want for saving videos or dumping cookies. The shared `browser` and the default-built `context` are still open; only your custom teardown decides when to close anything you opened yourself.

For deeper customization, such as a persistent context from scratch, override `context` instead and the default `page` will pick up whatever context you produce. Note that an Rstest fixture override cannot consume the value it overrides: a custom `page` builds from `context`, a custom `context` builds from `browser`.

## Personas and composed fixtures

`test.extend` composes, so you can layer fixtures the same way you'd layer options:

```ts
const test = base.extend<{
  adminAgent: PlaywrightAgent;
}>({
  url: 'http://localhost:5173/',
  adminAgent: async ({ browser }, use) => {
    const context = await browser.newContext({ storageState: 'admin.json' });
    const page = await context.newPage();
    await page.goto('http://localhost:5173/');
    const agent = new PlaywrightAgent(page);
    await use(agent);
    await agent.destroy();
    await context.close();
  },
});

test('only admin can delete', async ({ agent, adminAgent }) => {
  await agent.aiAssert('no Delete button is visible');
  await adminAgent.aiAssert('a Delete button is visible');
});
```

This is the recommended pattern for multi-persona tests, mock-server fixtures, database seeding, and anything else you'd otherwise reach for `beforeEach` to do. Fixtures compose by dependency graph and tear down in reverse, so ordering is explicit and correct.
