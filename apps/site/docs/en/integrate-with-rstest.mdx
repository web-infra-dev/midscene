# Integrate with Rstest

[Rstest](https://github.com/web-infra-dev/rstest) is a fast, Rspack-based testing framework. With `@midscene/rstest`, you can describe browser interactions and assertions in natural language and run them as regular Rstest tests:

```ts
import { expect, test } from '@midscene/rstest/playwright';

test('adds a todo', async ({ agent, page }) => {
  await page.goto('http://localhost:5173/');
  await agent.aiAct("type 'Study AI' in the input and press Enter");
  await agent.aiAssert('the list contains exactly one item: "Study AI"');
});
```

Midscene launches the browser, creates an agent for each test, and generates a visual HTML report for every test file, so you can replay each AI step with screenshots.

Since `@midscene/rstest` is built on [`@rstest/playwright`](https://github.com/web-infra-dev/rstest/tree/main/packages/playwright), everything it offers is also available here: Playwright-flavored `expect` (`toBeVisible`, `toHaveText`, etc.), debug mode (`PWDEBUG`), trace capture, and the `request` / `serve` fixtures.

:::info Playwright only

`@midscene/rstest` drives the browser through Playwright, which is why the import path is `@midscene/rstest/playwright`. Puppeteer is not supported.

:::

## Step 1: Install dependencies

Install `@midscene/rstest` together with `@midscene/web`, `@rstest/core`, `@rstest/playwright`, and `playwright`:

<PackageManagerTabs command="install @midscene/web @midscene/rstest @rstest/core @rstest/playwright playwright --save-dev" />

:::tip Version requirements

- `@rstest/core >= 0.11.2`
- `playwright >= 1.49.1`
- Node.js `^20.19.0 || >=22.12.0`

:::

## Step 2: Configure Rstest

Create `rstest.config.ts` in the root of your test project:

```ts title="rstest.config.ts"
import 'dotenv/config'; // load model env vars from .env
import MidsceneReporter from '@midscene/rstest/reporter';
import { defineConfig } from '@rstest/core';

export default defineConfig({
  include: ['e2e/**/*.test.ts'],
  // AI calls take time, so leave plenty of headroom for tests and hooks.
  testTimeout: 1_800_000,
  hookTimeout: 60_000,
  reporters: ['default', new MidsceneReporter()],
});
```

Register `MidsceneReporter` in `reporters`. After a test file finishes, it merges the reports of all tests in that file into one HTML report and prints the path to the console.

## Step 3: Write a test

Import `test` and `expect` from `@midscene/rstest/playwright`, then destructure the fixtures you need in the test function:

```ts title="e2e/todo-list.test.ts"
import { expect, test } from '@midscene/rstest/playwright';

test('adds and completes a todo', async ({ agent, page }) => {
  await page.goto('http://localhost:5173/');
  await agent.aiAct("type 'Study AI' in the input and press Enter");
  await agent.aiAct('click the checkbox of the first item');

  const list = await agent.aiQuery<string[]>(
    'string[], the complete task list',
  );
  expect(list).toHaveLength(1);

  await agent.aiAssert('the only item is marked as done');
});
```

Call `page.goto(...)` at the start of the test to open the page you want to test, then use `agent` to interact with it. See the [API Reference](./api#interaction-methods) for all agent methods.

The available fixtures:

| Fixture | Description |
| --- | --- |
| `agent` | The Midscene agent for the current page. Use `agent.aiAct(...)`, `agent.aiQuery(...)`, `agent.aiAssert(...)`, etc. to drive and check the page. |
| `page` | The Playwright `Page`. Use it to navigate (`page.goto`) and to call any Playwright API, such as `page.route(...)` or `page.evaluate(...)`. |
| `browser` | The shared Playwright `Browser`. It launches when first used and closes automatically. Use it to open extra contexts or pages. |
| `context` | The `BrowserContext` that `page` belongs to. |
| `request` / `serve` | From `@rstest/playwright`: an isolated `APIRequestContext` for HTTP calls, and a one-line static file server. |
| `agentForPage(page, opts?)` | Creates a Midscene agent for another page, such as a popup or a second session. Its steps appear in the same report. |
| `playwright` | Browser options: launch, context, debug, trace. See [Configure the browser](#configure-the-browser). |
| `agentOptions` | Agent options: cache, action context, and more. See [Configure the agent](#configure-the-agent). |

AI methods and Playwright APIs mix freely. For example, mock an API with `page.route` and verify the result with `aiAssert`:

```ts
test('mocks the API and verifies the UI', async ({ agent, page }) => {
  await page.route('**/api/contacts', (route) =>
    route.fulfill({ json: [{ name: 'Fake Alice' }] }),
  );
  await page.goto('http://localhost:5173/');
  await agent.aiAssert('the contact list shows "Fake Alice"');
});
```

## Step 4: Run the tests

```bash
npx rstest run

# or, while iterating:
npx rstest watch
```

After each test file finishes, the console prints the report path:

```text
Midscene report: midscene_run/report/<file>.html
```

Open it in your browser to replay every AI step with screenshots. To process report files programmatically, see [Consume report files](./consume-report-file).

## Configure the browser

Browser behavior — launch options, context options, debug, trace — is configured through the `playwright` fixture, which comes from `@rstest/playwright`. Every option in the [upstream docs](https://github.com/web-infra-dev/rstest/tree/main/packages/playwright) is available:

```ts
import { test as base } from '@midscene/rstest/playwright';

const test = base.extend({
  playwright: {
    launchOptions: { headless: false },
    contextOptions: { locale: 'zh-CN', storageState: 'auth.json' },
    debug: true, // headed + slowMo + DevTools, pause on failure (also: PWDEBUG=1)
    trace: 'retain-on-failure', // save trace.zip (also: RSTEST_PLAYWRIGHT_TRACE)
  },
});
```

`@midscene/rstest` ships a default value for this fixture — headless in CI, headed locally, with a 1920×1080 viewport. Overriding the fixture replaces the whole object, so spread `defaultPlaywrightOptions` to keep the defaults:

```ts
import {
  test as base,
  defaultPlaywrightOptions,
} from '@midscene/rstest/playwright';

const test = base.extend({
  playwright: {
    ...defaultPlaywrightOptions,
    contextOptions: { viewport: { width: 375, height: 667 } },
  },
});
```

Saved traces land in `.rstest/playwright-traces/`, and the console prints a `npx playwright show-trace ...` command to open them.

## Configure the agent

The `agentOptions` fixture sets the options used to construct every agent in a test. It accepts the same options as `PlaywrightAgent`, such as `aiActionContext`, `modelConfig`, `waitAfterAction`, and `replanningCycleLimit`:

```ts
const test = base.extend({
  agentOptions: {
    aiActionContext: 'Close any promo dialog first.',
    cache: true,
  },
});
```

Set `cache: true` to reuse AI planning results across runs and speed up repeated executions. Midscene derives a stable cache id from the test file name and test name, so re-running the same test hits the same cache. To control the strategy or the id yourself, pass `cache: { strategy, id }`. See [Caching](./caching).

`agentForPage(page, opts)` accepts the same options per call; they take precedence over `agentOptions`.

## Set project-wide defaults

To apply the same options to every test file, extend the base test in a shared module and import your extended `test` everywhere:

```ts title="tests/midscene.ts"
import {
  test as base,
  defaultPlaywrightOptions,
} from '@midscene/rstest/playwright';

export { describe, expect } from '@midscene/rstest/playwright';

export const test = base.extend({
  playwright: {
    ...defaultPlaywrightOptions,
    contextOptions: { locale: 'en-US', viewport: { width: 1280, height: 720 } },
  },
  agentOptions: { cache: true },
});
```

```ts title="tests/todo.test.ts"
import { expect, test } from './midscene';
```

A file that needs different values calls `extend` again — the later value replaces the earlier one.

## Customize the page

To open the same URL for every test in a file, or to run setup that needs the page before the test starts — init scripts, storage state, and so on — override the `page` fixture. Code before `await use(page)` runs before each test; code after runs as teardown:

```ts
import type { Page } from 'playwright';
import { test as base } from '@midscene/rstest/playwright';

const test = base.extend<{ page: Page }>({
  page: async ({ context }, use) => {
    await context.addInitScript(() => {
      window.localStorage.setItem('feature-flag', 'on');
    });
    const page = await context.newPage();
    await page.goto('https://example.com');

    await use(page);

    await page.close();
  },
});
```

When your teardown code runs, Midscene has already collected the report and the page is still open — a good place to save videos or export cookies.

For deeper customization, such as a persistent context, override `context` instead; the default `page` is created from whatever context you provide. Note that a fixture override cannot read the value it replaces: build a custom `page` from `context`, and a custom `context` from `browser`.

## Test with multiple pages or users

Use `agentForPage` to get an agent for any page beyond the primary one. Its steps are recorded in the same report, and cleanup is automatic.

Drive a popup:

```ts
test('drives a popup with a second agent', async ({
  agent,
  page,
  agentForPage,
}) => {
  await page.goto('http://localhost:5173/');
  await agent.aiTap("the 'View details' button");
  const popup = await page.waitForEvent('popup');

  const popupAgent = await agentForPage(popup);
  await popupAgent.aiAssert('the popup shows order details');
});
```

Run two logged-in users in one test — open a second browser context (with its own cookies and storage) from `browser`:

```ts
test('user B receives the message from user A', async ({
  agent,
  page,
  browser,
  agentForPage,
}) => {
  await page.goto('http://localhost:5173/');

  const sessionB = await browser.newContext({ storageState: 'user-b.json' });
  const pageB = await sessionB.newPage();
  await pageB.goto('http://localhost:5173/');
  const agentB = await agentForPage(pageB);

  await agent.aiAct("send 'hello' to user B");
  await agentB.aiWaitFor("the chat shows a new message 'hello'");
});
```

## Reuse setup with custom fixtures

In Rstest, hooks such as `beforeEach` cannot access fixtures. Shared setup — logging in, seeding data, preparing a second role — is written as a custom fixture via `test.extend`, and your fixtures can depend on `agent` and `agentForPage`.

Log in before every test in a file — declare the fixture with `{ auto: true }` so it runs even when tests don't reference it:

```ts title="e2e/fixtures.ts"
import { test as base } from '@midscene/rstest/playwright';

export const test = base.extend<{ loggedIn: void }>({
  loggedIn: [
    async ({ page, agent }, use) => {
      await page.goto('http://localhost:5173/');
      await agent.aiAct('log in with alice / password');
      await use(undefined);
    },
    { auto: true },
  ],
});

export { expect } from '@midscene/rstest/playwright';
```

Add an always-available second role, with its own login state:

```ts
import type { PlaywrightAgent } from '@midscene/web/playwright';

const test = base.extend<{ adminAgent: PlaywrightAgent }>({
  adminAgent: async ({ browser, agentForPage }, use) => {
    const context = await browser.newContext({ storageState: 'admin.json' });
    const page = await context.newPage();
    await page.goto('http://localhost:5173/');
    await use(await agentForPage(page));
    await context.close();
  },
});

test('only admin can delete', async ({ agent, page, adminAgent }) => {
  await page.goto('http://localhost:5173/');
  await agent.aiAssert('no Delete button is visible');
  await adminAgent.aiAssert('a Delete button is visible');
});
```

`agent` and `agentForPage` themselves cannot be overridden in `test.extend` — TypeScript reports an error, because a replacement would lose report collection. Depend on them from your own fixtures instead, as above.

## Configure the model

Model access is configured through environment variables — API endpoint, key, model name, and more. Set them in `.env`, CI secrets, or shell exports, and they take effect automatically. See [Model configuration](./model-config) for the full list.

```bash
MIDSCENE_MODEL_BASE_URL=https://api.openai.com/v1
MIDSCENE_MODEL_API_KEY=sk-...
MIDSCENE_MODEL_NAME=qwen3.7-plus
```

To change model configuration at runtime — for example, switching models based on the environment — call `overrideAIConfig` in a setup file:

```ts title="midscene.setup.ts"
import { overrideAIConfig } from '@midscene/rstest/playwright';

overrideAIConfig({
  MIDSCENE_MODEL_NAME: process.env.CI ? 'gpt-5.5' : 'gpt-5.4',
});
```

To print debug logs, use the `DEBUG` environment variable:

```bash
DEBUG=midscene:*,rstest:* npx rstest run
```
